# 产品文案规范

> 中文优先，短句优先。PicNexus 是高频工具，文案要帮用户判断状态、完成动作，不抢注意力。

## 口吻原则

- 直接：先说发生了什么，再说用户能做什么。
- 克制：界面已经表达的结果，不再用 Toast 重复一次。
- 稳定：同类状态用同一套词，降低用户每次重新理解的成本。
- 诚实：危险操作必须说明影响范围和是否可撤销。
- 专业但不绕：保留「图床」等域内词，首次使用或空状态里补一句自然解释。

## 统一句式

| 场景 | 模板 | 示例 |
|---|---|---|
| 成功态 | 已 + 动词 | 已上传、已下载、已同步 |
| 失败态 | 动作失败 | 上传失败、复制失败、恢复失败 |
| 失败详情 | 原因 + 下一步 | 网络超时。请检查图床配置后重试 |
| 阻塞态 | XX 进行中 | 上传进行中、扫描进行中 |
| 空状态 | 当前状态 + 下一步 | 暂无可用图床。请在设置中配置图床 |
| 危险确认 | 会发生什么 + 是否可撤销 | 确定要清空上传队列吗？此操作不可撤销。 |

## 通道选择

- Toast：用户主动操作后的轻量回执，遵循 `docs/design/notification-patterns.md`。
- Confirm：删除、覆盖、清空、恢复默认等需要用户决策的动作。
- Dialog：需要输入、多步选择或信息密度高的流程。
- Inline banner：需要用户稍后处理、不能自动消失的状态，例如刷新、重启。
- Silent：单条复制、内部状态记录等高频低风险反馈。

## 文案资产

- 通用可见文案集中在 `src/constants/uiCopy.ts`。
- Toast 文案继续通过 `src/constants/toastMessages.ts` 暴露，新增 Toast 优先复用 `UI_COPY` 中的模板。
- 动态文案使用函数模板，不在组件里拼接长句。
- 新增生产可见中文文案时，优先加入 `UI_COPY`；确需局部保留时，在文案审计清单中标注原因。

## 反模式

- 不写「操作成功！」这类空泛反馈，改成「已上传」「已同步」。
- 不把技术错误原样丢给用户，除非它是唯一可执行线索；必要时加下一步。
- 不在按钮已经禁用、列表已经清空、进度条已经显示时再弹 Toast。
- 不在危险按钮上使用含糊文案，例如「确定」「继续」；改成「删除」「清空」「覆盖」。
